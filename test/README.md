# ProfSync Test Stack

A local Docker environment that runs Sonarr and Radarr instances pre-loaded with test fixtures. It exists so you can run the ProfSync wizard end-to-end — creating custom formats and quality profiles — without touching a real media server. The stack handles container setup, API key extraction, root folder creation, fixture import, and post-deployment validation automatically.

## Quick Start

```bash
cd test/
./init-stack.sh
```

This will:
1. Start Sonarr and Radarr containers
2. Wait for both to be ready
3. Extract and validate API keys from their config files
4. Configure root folders
5. Import test libraries (20 series, 20 movies)
6. Create `.env` with connection details
7. Run `validate-stack.sh` to confirm everything is healthy

## Running the Wizard

After initialization, run the wizard with:

```bash
cd ..
python wizard.py --teststack
```

In `--teststack` mode, Sonarr and Radarr connection details are auto-loaded from `test/.env` — no need to enter URLs and API keys manually.

Or run manually:

```bash
python wizard.py
# When prompted:
# Sonarr URL: http://localhost:8989
# Sonarr API Key: [from test/.env]
# Radarr URL: http://localhost:7878
# Radarr API Key: [from test/.env]
```

## Validation

Two scripts verify the stack and wizard results:

**`validate-stack.sh`** — runs automatically at the end of `init-stack.sh`. Checks both Sonarr and Radarr for:
- API responds HTTP 200 with a valid API key
- Root folders exist
- Fixtures imported (series/movie count > 0)
- Quality profiles and custom formats endpoints accessible

**`validate-wizard.sh`** — run after the wizard completes. Checks both Sonarr and Radarr for:
- ProfSync custom formats exist
- ProfSync quality profile exists
- Profile has qualities with `allowed=true`
- Profile has format items with non-zero scores
- Profile cutoff is set

```bash
cd test/
./validate-wizard.sh
```

**`run-wizard-noninteractive.py`** — runs the wizard without an interactive terminal (for CI or scripted testing). Uses hardcoded balanced defaults and calls the same code paths as `wizard.py --teststack`.

```bash
cd test/
python run-wizard-noninteractive.py
./validate-wizard.sh
```

## Authentication

The test stack runs with **authentication disabled** for simplicity:
- Web UIs accessible without login at http://localhost:8989 (Sonarr) and http://localhost:7878 (Radarr)
- API calls only require the `X-Api-Key` header
- Safe for local testing only — do not use this config in production

## Files

| File | Purpose |
|------|---------|
| `docker-compose.yml` | Sonarr + Radarr container definitions |
| `init-stack.sh` | Bootstrap and configure services |
| `cleanup-stack.sh` | Stop and remove containers |
| `validate-stack.sh` | Post-deployment health checks |
| `validate-wizard.sh` | Post-wizard correctness checks |
| `run-wizard-noninteractive.py` | CI-friendly wizard runner |
| `import-fixtures.sh` | Imports series/movies from `fixtures/` |
| `fixtures/` | JSON fixture data (20 series, 20 movies) |
| `.env` | Generated by init-stack.sh (API keys, URLs) |
| `sonarr-config/`, `radarr-config/` | Service configuration volumes |
| `sonarr-tv/`, `sonarr-downloads/` | Sonarr media/staging dirs |
| `radarr-movies/`, `radarr-downloads/` | Radarr media/staging dirs |

## Cleanup

```bash
cd test/
./cleanup-stack.sh
```

Stops containers and optionally removes data volumes.

For a full reset (nuke everything and start fresh):

```bash
cd test/
docker-compose down -v
rm -rf sonarr-config radarr-config sonarr-tv radarr-movies sonarr-downloads radarr-downloads .env
./init-stack.sh
```

## Troubleshooting

**Services won't start:**
```bash
docker-compose logs sonarr
docker-compose logs radarr
```

**API key extraction fails:**
- Wait longer for services to initialize (check logs)
- Manually extract from config files:
  - `sonarr-config/config.xml` (look for `<ApiKey>`)
  - `radarr-config/config.xml` (look for `<ApiKey>`)

**Port conflicts:**
- Change ports in `docker-compose.yml` if 8989 or 7878 are in use
- Update corresponding URLs in `.env` after initialization
